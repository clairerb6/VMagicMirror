# URP移行タスク

## 現状整理

- Unity プロジェクト全体としては URP で起動できており、アバターのロードと通常動作は確認済み。
- アバターには URP の MToon が割り当てられているように見え、シェードも効いていることを確認済み。
- アバターのアウトライン表示、ポスプロ適用、透過出力、透過出力時の切り抜き、SSAO、アンチエイリアス設定は確認済み。
- アクセサリーのマテリアル割り当て、とくに MToon 系の割り当ては確認済み。
- サブキャラのマテリアル割り当ても確認済み。
- キーボード / マウスなどのデバイスオブジェクト表示、スクリーンショット機能も確認済み。
- 一方で、固定影の板ポリ描画と品質ごとの URP Asset 切り替えは未解決または要確認。

## 現時点の未解決タスク

### P0: 品質設定ごとの URP Asset 定義を固める

- `ProjectSettings/QualitySettings.asset` における品質ごとの `customRenderPipeline` の実運用方針を決める。
- 単一の URP Asset を全品質で使うか、品質別に URP Asset を分けるかを決める。
- スクリプトで差し替える場合も含め、実際の切り替え経路を明示する。

### P1: 固定影 (ShadowDrawer) を確認・復旧する

- 陰を板ポリに描く機能が URP で正しく動くか確認する。
- `MainLight` ベースの通常 shadow とは分離し、`ShadowLight` 視点の専用 shadow map を `ShadowBoard` 用に生成する方針で進める。
- 受け側は「投影マスクを貼る」のではなく、`ShadowBoard` の各ピクセルを `ShadowLight` 空間へ変換して shadow map と深度比較する方式を採る。
- caster を「アバター限定」で決め打ちする方針は採らず、距離ベースの除外ルールで shadow map に描く対象を決める。
  - 近傍の renderer はアバター、アクセサリー、必要ならサブキャラも自然に含める。
  - 遠景オブジェクトは `maxCasterDistance` 相当のルールで除外する。
  - layer は補助的に使えてもよいが、主たる選別ルールは距離と除外条件に寄せる。
- 実装メモ:
  - いったん `ProjectedShadow...` 系の「投影マスク方式」は残してあるが、本命は `ShadowMap...` 系の「専用 shadow map + 深度比較」方式。
  - `ShadowLight` の向き自体は新実装に反映できている。問題は「向きが反映されない」ことではなく、向きによって影が不自然に長く伸びたり、一部が切れたりする点。
  - `forward / -forward` の自動 score 選択は主因ではなかった。`forward` 固定では影が消え、`-forward` 固定で従来の見え方に戻るため、現状の light-space 解釈では `-shadowLight.transform.forward` を採るほうが整合している。
  - 受け面を caster 全体に tight fit するより、`ShadowBoard` 基準で安定した light-space を作り、caster は主に `near / far` 方向で収めるほうが `ShadowLight` の回転追従性は良い。
  - `ShadowLight` が浅い角度になると影が長く伸びるのは、板ポリへの投影としては幾何的に自然な挙動。必要なら pitch/yaw の制限、浅い角度でのフェード、影長 clamp などの見た目補正を検討する。
  - 「真下向きに近い + yaw 180 / -180 では頭まで影が出るが、yaw 0 付近だと腰あたりで切れる」現象については、shadow map 生成側だけでなく `FixedShadow/ShadowBoard` 自体のジオメトリ制約が本命。
    - `FixedShadow.prefab` の `ShadowBoard` は床ではなく原点付近に立つ 2x2 の縦板であり、受け側 shader はその板ポリ上の画素にしか影を描けない。
    - そのため、`ShadowLight` の yaw によっては頭部の投影先が板の上端を越え、shadow map に頭まで入っていても `ShadowBoard` 上では腰あたりで切れて見える可能性が高い。
    - 次回確認時は、一時的に `FixedShadow/ShadowBoard` の高さや位置を変えたときに切れ方が変わるかを見ると、受け面サイズ起因かどうかを素早く切り分けられる。

### P1': 固定影の代替としてドロップシャドウ系ポスプロを検討・実装する

- メジャーケースでは「アバターの背面に自然な影が見える」ことが重要であり、物理ベースのライティング shadow を厳密に再現する必要は薄い。
- そのため、`ShadowDrawer` / `ShadowBoard` の板ポリ投影をそのまま復旧する代わりに、Post Process の一種として擬似的なドロップシャドウを描く案を優先検討する。
- 基本方針:
  - 描画済み画像のうち `alpha > 0` の領域をアバターのシルエット近似として扱う。
  - そのシルエットをスクリーン空間またはそれに準じる空間でオフセットし、必要に応じてスケールして「影領域」を作る。
  - 生成した影領域をアバターの背面側に重ねるように alpha blend し、既存の透明表現を壊さないようにする。
  - 実装初期段階では UI や設定画面とは結合せず、Volume で決め打ちパラメータを与えて動作確認できるようにする。
- 要件:
  - Volume ベースで ON/OFF できる独自 Post Process として実装する。
  - 少なくとも「影のオフセット」と「影のスケール」に相当するパラメータを持たせる。
  - 将来的にライト方向や距離感の近似へつなげられるよう、オフセット量と拡大量は個別に調整可能にする。
  - 既存の alpha を考慮したブレンドを行い、透過出力や切り抜きとの整合を保つ。
  - 「アバターの背面にある影」として見えることを優先し、必要なら深度や描画順を利用して前面への不自然なかぶりを避ける。
- 実装メモ:
  - まずは単純なシルエット複製 + offset/scale + 半透明合成で成立するか確認する。
  - ライト方向の近似は、初期実装では Volume の固定パラメータで与え、UI 連携は後回しにする。
  - 既存の `ShadowDrawer` を直ちに削除する必要はないが、URP 向けの実運用候補としてはこちらを優先して評価する。
  - 実装候補は大きく 2 つある:
    - 最速試作案: fullscreen shader 1 回で、最終カラーの `alpha > 0` 領域をそのまま shadow 元シルエットとして扱い、offset / scale して背面っぽく合成する。
    - 最有力案: アバター renderer 群だけを別 RT にマスク描画し、そのマスクを fullscreen 合成する 2 段構成にする。
  - 各案の長所と短所:
    - 最速試作案の長所: 計算量が軽い。fullscreen 1 pass で成立し、既存 URP 構成への差し込みも比較的容易。
    - 最速試作案の短所: アバター以外の `alpha > 0` 領域も shadow 化しうる。表示物によっては不要な影が混ざる。
    - 最有力案の長所: アバター限定マスクを使えるため、影の対象を厳密に制御しやすい。最終的な品質・運用面ではこちらが本命。
    - 最有力案の短所: マスク生成 pass が増えるため、fullscreen 1 pass 案より計算量が重い。実装も一段複雑。
  - 方針決定:
    - 計算量を抑えて見栄え評価を先に回したいモチベーションが高いため、まずは最速試作案を実装して検証する。
    - その結果、アバター以外の影混入や制御性の不足が問題になる場合に、最有力案へ段階的に進める。
